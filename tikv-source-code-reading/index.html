TiKV 读写流程源码阅读

本文最后更新于：10 小时前

背景

TiKV 是一个支持事务的分布式 Key-Value 数据库，目前已经是 CNCF 基金会 的顶级项目。

想要参与到 TiKV 的研发中来，首先需要学习 Rust 语言，其次需要理解 TiKV 的原理，最后便是在前两者的基础上去阅读学习 TiKV 的 codebase。

本人在 PingCAP 事务组实习的过程中，首先入门了 Rust 语言，接着又基于之前的一些知识系统地学习了 TiKV 事务层的演示历史，最后开始尝试学习 TiKV 的 codebase 并尝试在其中做一些探索性的预研工作。

TiKV 的 官方源码解析文档 详细的介绍了 TiKV 3.x 版本不同模块的设计要点，主要流程和相应代码片段，是学习 TiKV 源码必读的学习资料。然而，TiKV 目前已经发展到了 6.x 版本，不仅引入了很多新的功能和优化，而且对源码也进行了多次重构，这使得一些官方源码阅读报告中的代码已经不复存在，无法与博客形成对照；此外尽管 TiKV 官方源码解析文档系统地介绍了不同模块的工作，但并没有将完整的读写流程串起来去介绍经过的模块和对应的代码片段，而对于新人来讲，尽快的熟悉读写流程的总链路会更便于从全局角度理解代码。

基于以上动机，本篇博客将首先介绍 TiKV 中的一些基本概念，接着以源码的视角介绍 TiKV 的 RPC 启动流程和读写流程，最后再简单介绍断点调试 TiKV 学习源码的方案。

希望本篇博客能够帮助对 TiKV 开发感兴趣的同学尽快了解 TiKV 的代码框架。

注：以下源码阅读基于 TiKV 6.1 版本。

基本概念

TiKV 的架构简介可以查看 官方文档。总体来看，TiKV 是一个支持 Multi-Raft 的分布式 KV 存储。

TiKV 的每个进程拥有一个 store，store 中拥有若干 region。每个 region 是一个 raft 组，会存在于副本数个 store 上管理一段 KV 区间的数据。

RPC 启动流程

本小节将介绍 TiKV 的 RPC 启动流程，并顺带介绍 TiKV 若干模块和结构的初始化过程，之后的章节将直接基于读写流程对应的 RPC handler 进一步介绍。

cmd/tikv-server/main.rs 是 TiKV 进程启动的入口，其主要做了以下两个工作：

  • 配置参数解析
  • 使用 server::server::run_tikv(config) 启动 tikv 进程

对于 server/src/server.rs 的 run-tikv 函数，其会调用 run_impl 函数并根据配置参数来启动对应的 KV 引擎。

在 run_impl 函数中，首先会调用 TikvServer::<CER>::init::<F>(config) 函数做若干重要模块的初始化，包含但不限于 batch_system, concurrency_manager,background_worker, quota_limiter 等等，接着在 tikv.init_servers::<F>() 里将 RPC handler 与 RPC server 绑定起来，最后在 tikv.run_server(server_config) 中便会使用 TiKV 源码解析第八章 介绍的 grpc-rs 绑定对应的端口并开始监听连接了。

RPC 服务启动后，所有发往 RPC 的请求便可以在 src/server/service/kv.rs 中找到对应的 handler，之后直接阅读对应的实现即可。有关 TiKV 目前支持 idea 接口，可以直接查看 kvproto 对应的 service Tikv，目前的 RPC 接口已经接近 60 个，每个 RPC 接口在代码中都会对应一个 handler。

回到 RPC handler 的具体实现中，与 TiKV 源码解析第七章 中介绍的 handler example 不同，当前 kv.rs 对部分 RPC handler 例如 prewrite, commit 等事务相关接口进行了重构，所有相关的逻辑被封装到了 handle_request 宏中，对应不同接口的实现被放到了 future_prewirte, future_commit 等 future_xxx 函数中，不过仍然有部分接口例如 coprocessor 等采用了原生对接 grpc-rs 的方式。

对于事务相关的 future_xxx，由于它们会被统一调度到 storage 结构体中（有关该 storage 是如何初始化的相信您已经能够从前面的启动流程中自行根据调用栈反向找到对应的初始化逻辑），当前又抽象出了 txn_command_future 宏来减少事务相关 future_xxx 函数的冗余代码。

自 3.0 版本以来，kv.rs 利用了多个宏显著减少了 RPC 接口的冗余代码，然而这些宏目前还不能被 Clion 等调试工具的函数调用关系链所检索到，这可能会困惑刚开始查看函数调用链却无法找到对应 handler 的新同学。

通过本小节，希望您能够简单了解 TiKV 启动流程中的若干重要步骤，具备寻找全局重要结构体的初始化代码片段的能力，此外也能够在需要时迅速找到需要的 RPC handler 去从上到下追踪一个 RPC 请求的具体流程。
